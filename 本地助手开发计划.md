# 本地助手开发计划

## 1. 项目目标

金凤 BT 当前上传门槛高：用户需要手动制作 torrent、上传、重新下载站点生成的最终 torrent，再把最终 torrent 指向原文件开始做种。纯网页又无法可靠获取拖入文件的绝对路径，因此无法直接解决“原地做种”的问题。

本地助手的目标是把上传体验压缩成：

1. 用户选择文件或文件夹。
2. 助手自动预填标题、简介、分类和标签。
3. 用户检查并修改信息。
4. 用户点击“一键做种”。
5. 助手自动生成 torrent、上传站点、获取最终 torrent，并调用 µTorrent 3.1 开始做种。

第一版优先解决桌面端上传者的问题；移动端浏览、PWA、App、服务器 seedbox 暂不进入本地助手 MVP。

## 2. 技术选择

- 开发语言：Python 3.11+
- 桌面 UI：PySide6
- BT 客户端：µTorrent 3.1
- HTTP 自动化：requests / httpx
- HTML 解析：BeautifulSoup4 / lxml
- torrent 编解码：bencodepy 或自写最小 bencode 模块
- 打包：PyInstaller
- 平台优先级：Windows 优先，后续再考虑 macOS/Linux

选择 Python + PySide6 的原因：开发速度快、文件系统能力完整、适合快速验证站点表单自动化和 µTorrent WebUI/命令行调用。

## 5. 核心流程设计

### 5.1 资源草稿

用户选择文件或文件夹后，助手生成资源草稿：

- 本地路径
- 资源名称
- 总大小
- 文件列表
- 标题
- 副标题
- 简介
- 分类
- 标签
- 是否确认内容合规

资源草稿只保存在本地，不上传原始文件内容。

### 5.2 生成 torrent

生成 torrent 时需要注意：

- 设置 private flag。
- 写入站点 Tracker。
- 保持文件名和目录结构稳定。
- 对大文件显示进度，避免用户误以为卡死。
- 生成后先保存到本地临时目录。

需要验证站点是否会改写 torrent。如果最终 torrent 的 info hash 与本地生成 torrent 不一致，则 µTorrent 做种逻辑必须以站点最终 torrent 为准，并确保它指向同一份本地文件。

### 5.3 上传站点

上传页面是 ASP.NET WebForms，提交时通常需要带上：

- `__VIEWSTATE`
- `__VIEWSTATEGENERATOR`
- `__EVENTVALIDATION`
- 登录 Cookie
- 资源标题字段
- 副标题字段
- 简介字段
- 分类字段
- 标签字段
- torrent 文件字段

实现时不要硬编码一次抓到的隐藏字段值；每次上传前都应 GET 上传页，解析最新隐藏字段，再 POST 表单。

### 5.4 获取最终 torrent

上传成功后，站点会跳转到资源详情页或返回上传结果页。助手需要：

- 解析上传是否成功。
- 找到详情页链接。
- 找到最终 torrent 下载链接。
- 下载最终 torrent。
- 保存到本地目录。
- 可选：对比本地 torrent 与最终 torrent 的 announce、private、info hash 差异。

### 5.5 调用 µTorrent 3.1

优先探索两种方式：

1. **命令行/文件关联方式**
   - 打开最终 torrent。
   - 用户或 µTorrent 使用默认下载目录。
   - 稳定性高，但不一定能指定原路径。

2. **µTorrent WebUI**
   - 通过本地 WebUI 添加 torrent。
   - 传入保存目录。
   - 适合自动化，但需要用户开启 WebUI 并配置账号密码。

由于用户不愿移动大型文件，MVP 应尽量支持指定原文件路径。如果 µTorrent 3.1 WebUI 无法稳定对任意路径做种，需要在助手中明确显示“当前客户端限制”，并保留手动选择路径的兜底流程。

- 选择文件

- 当前选择路径

- 标题输入框
- 副标题输入框

- 合规确认复选框
- 一键做种按钮
- 日志输出区域

### 设置页面

- 金凤站点地址
- Tracker 地址
- 用户名
- 密码或 Cookie 导入
- µTorrent 安装路径
- µTorrent WebUI 地址
- µTorrent WebUI 用户名
- µTorrent WebUI 密码
- 临时文件目录
- 最终 torrent 保存目录

账号密码和 Cookie 不应提交到仓库，后续应存储在用户本地配置或系统凭据管理中。

## 8. 验收标准

MVP 完成时应满足：

- 用户不需要手动制作 torrent。
- 用户不需要手动填写大部分资源信息。
- 用户不需要手动找上传后的最终 torrent。
- 用户原始文件不需要移动到专门下载目录，除非 µTorrent 3.1 的技术限制无法绕开。
- 上传失败时能看到可理解的错误原因。
- 不会把账号、Cookie、本地路径、下载内容提交进 git。

## 9. 风险与注意事项

### 合规风险

助手必须在上传前展示内容规范，并要求用户确认。建议保留举报入口说明，不鼓励上传违法违规或未授权传播内容。

### 账号安全

不要把账号密码、Cookie、WebUI 密码写入代码或提交仓库。调试日志必须隐藏敏感字段。

### µTorrent 3.1 风险

µTorrent 3.1 较旧，WebUI 和路径控制能力有限。它适合作为当前站点兼容方案，但后续应保留客户端适配层，方便换成 qBittorrent、Transmission 或其他客户端。

### 站点表单变化

上传页使用 ASP.NET WebForms，字段名和隐藏字段可能变化。代码应集中封装表单解析逻辑，避免字段散落在 UI 代码中。

### 大文件体验

生成 torrent 需要读取文件并计算分片哈希，大文件可能耗时。UI 必须显示进度，并允许用户取消。

## 10. 后续接手建议

1. 先做命令行原型，不要一开始写复杂 UI。
2. 先跑通一个小文件完整闭环。
3. 再测试文件夹和大文件。
4. 每发现一个站点或客户端限制，都写进 README 或 issue。
5. 功能稳定后再美化 UI。
6. 尽早找下一届维护者试用，避免只在开发者电脑上可用。
