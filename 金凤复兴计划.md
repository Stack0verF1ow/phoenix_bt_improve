## 金凤复兴计划

> 目标：在毕业前把金凤 BT 从“会用的人才会用”改造成“新生看一眼也敢点、手机上也能逛、上传者愿意持续做种”的校园资源分享平台。

### 一、核心判断

1. **不要一开始就做完整原生 App。** 先做移动端网页/PWA，把浏览、搜索、收藏、下载入口做好；后续如果活跃度回升，再封装 App。
2. **上传流程优先级最高。** 现在教程需要制作种子、上传、重新下载种子、再指向原文件做种，门槛太高。应先把流程压缩到“拖入文件/文件夹 → 自动生成种子 → 填信息 → 一键开始做种”。
3. **内容运营和产品改造要一起做。** 只改功能不运营，没人知道；只宣传不降门槛，新人还是会流失。
4. **合规边界必须前置。** 校园网资源分享要明确可分享内容、禁传内容、举报处理、管理员审核，避免项目刚复兴就因风险中断。

### 二、现状问题拆解

- 上传门槛高
  - 需要用户先理解 BT、Tracker、私有种子、做种路径等概念。
  - 当前上传页只让用户选择 `.torrent` 文件，不能拖拽，也不能自动制作种子。
  - 教程要求上传后再下载一次站点改写过的种子，很多新人会在这里放弃。
- 移动端体验弱
  - 手机/平板用户更适合浏览、收藏、复制下载链接，而不是制作和长期做种。
  - 没有 PWA/移动端首页，会损失大量“只想看看有什么资源”的潜在用户。
- 新人冷启动不足
  - 缺少“我为什么要用它”“我能在这里找到什么”“我怎么贡献”的入口。
  - 老教程偏技术流程，不像现代产品的新手引导。
- 社区感不足
  - BT 站点本质上依赖贡献者；如果没有激励、榜单、专题和反馈，上传者很难持续贡献。

### 三、产品方向建议

#### 1. 先做“无痛上传”而不是“直接上传大文件”

最现实的第一阶段不是把所有文件都传到服务器，而是让浏览器或本地小工具帮用户完成种子制作与站点提交：

- 拖拽文件或文件夹到上传页。
- 浏览器本地读取文件并生成 `.torrent`，文件本体不上传到服务器。
- 自动写入站点 Tracker、私有种子标记和用户必要信息。
- 自动从文件名提取标题、分类建议、标签建议。
- 上传完成后直接给出“最终做种种子”和“一键打开 BT 客户端”的按钮。
- 用清晰提示告诉用户：不要移动原文件，保持客户端开启，才能让别人下载。

如果以后学校资源、社团作品、公开授权资料有托管需求，再考虑“服务器临时收文件 + 校园 seedbox 自动做种”。这个方案体验最好，但会带来存储、带宽、审核、版权和滥用风险，不适合作为第一步。

#### 2. 移动端先做 PWA

PWA 比原生 App 更适合毕业前落地：

- 不需要应用商店审核。
- 电脑、安卓、iPad、iPhone 都能访问。
- 可以“添加到主屏幕”，看起来接近 App。
- 后续如有必要，可以用 WebView/Tauri/Capacitor 封装成 App。

移动端重点不是复杂上传，而是：

- 首页推荐。
- 分类浏览。
- 搜索和标签。
- 收藏/稍后下载。
- 一键复制磁力链接或下载 `.torrent`。
- 热门资源、最近活跃、长期有人做种资源优先展示。

#### 3. 把“资源站”做成“校园共同记忆”

可以主打合法、校园内有价值、能形成归属感的内容：

- 社团公开视频、讲座录像、校园活动资料。
- 公开课资料、学习资料、工具软件的官方安装包镜像。
- 学长学姐整理的课程经验包、竞赛资料包。
- 毕业季作品、摄影集、音乐作品、开源项目归档。
- 公共领域或明确授权传播的内容。

这样更容易获得学生会、社团、学院、校园网管理方的支持，也更容易长期存活。

### 四、毕业前最小可行路线

#### 阶段 0：调研与合规确认（1 周）

- [ ] 统计最近 3-6 个月访问量、上传量、下载量、活跃做种数。
- [ ] 找 5-10 个同学试用当前上传流程，记录他们在哪一步放弃。
- [ ] 整理现有站点权限：是否能改前端、后端、数据库、Tracker 配置。
- [ ] 明确管理员和校园网规则：哪些内容允许分享，哪些内容必须禁止。
- [ ] 写一版《上传内容规范》和《举报处理流程》。

#### 阶段 1：上传流程减负（2-4 周）

- [ ] 在上传页增加拖拽区域，支持拖入 `.torrent` 文件。
- [ ] 上传 `.torrent` 后自动解析名称、大小、文件列表，预填标题和简介。
- [ ] 增加上传前检查：是否私有种子、是否包含 Tracker、文件名是否异常。
- [ ] 上传完成页改成“下一步做种向导”，明确提示用户要下载最终种子并指向原文件。
- [ ] 提供 qBittorrent、Transmission、移动端 BT 客户端的新版简短教程。
- [ ] 把旧 13 步教程改成“30 秒版”“详细版”“常见错误版”。
- [ ] 如果后端允许，改成上传完成后直接返回最终种子，减少“自己再找下载链接”的步骤。
- [ ] 如果可行，开发浏览器端“选择文件夹 → 本地生成种子 → 自动上传”的实验版。

#### 阶段 2：移动端/PWA 改造（2-4 周）

- [ ] 重做首页为响应式布局，手机上优先显示搜索、分类、热门和最新。
- [ ] 增加 PWA 配置：站点图标、启动页、添加到主屏幕提示。
- [ ] 优化资源详情页：移动端突出下载按钮、文件列表、做种人数、上传时间。
- [ ] 增加收藏/稍后下载功能，适配手机浏览、电脑下载的使用场景。
- [ ] 增加“复制下载链接/复制磁力链接/下载种子”三个明确入口。
- [ ] 优化搜索结果：按有做种、热度、时间排序，减少点进死种的挫败感。
- [ ] 做一页“新手入口”：什么是 BT、怎么下、怎么上传、为什么要做种。

#### 阶段 3：内容运营与社区激励（持续）

- [ ] 每周做 1 个专题合集，例如“期末复习资料”“毕业季作品”“社团公开资源”。
- [ ] 设置上传者徽章：首传、连续做种、优质整理、社团贡献等。
- [ ] 做一个“本周有人做种榜”，避免只按下载量推荐老资源。
- [ ] 建立投稿群或反馈表，收集想要的资源、死种反馈、上传问题。
- [ ] 邀请社团和课程助教发布公开资料，先把供给端做起来。
- [ ] 在新生群、社团群、校园论坛发布“金凤新手指南”。
- [ ] 设立管理员/志愿者值班，处理举报、死种、分类错误和上传求助。

#### 阶段 4：进阶能力（有余力再做）

- [ ] 本地上传助手：自动生成种子、调用 BT 客户端、提交站点表单。
- [ ] 安卓分享入口：从文件管理器分享到金凤助手，自动生成上传草稿。
- [ ] 校园 seedbox：仅用于社团作品、公开资料等合规内容的自动做种。
- [ ] 资源健康度评分：结合做种数、完成数、最近活跃时间给出可下载性提示。
- [ ] API 化：为 PWA、App、上传助手提供统一接口。
- [ ] 站内通知：收藏资源恢复做种、专题更新、上传审核结果提醒。

### 五、优先级任务清单

#### P0：必须先做

- [ ] 取得站点维护权限或确认能改哪些部分。
- [ ] 和管理员确认内容规范、举报入口、封禁规则。
- [ ] 把上传页改成更清晰的向导式流程。
- [ ] 上传完成后给出明确的“下载最终种子并开始做种”按钮。
- [ ] 重写上传教程，优先支持 qBittorrent，减少过时软件和旧系统截图。
- [ ] 做移动端首页和详情页适配。
- [ ] 建立基础数据看板：日活、周活、上传数、有效做种数、移动端占比。

#### P1：显著提升体验

- [ ] 支持拖拽上传 `.torrent`。
- [ ] 自动解析种子文件并预填标题、简介、标签、分类。
- [ ] 增加资源健康度：有做种、低做种、疑似死种。
- [ ] 增加收藏/稍后下载。
- [ ] 增加专题页和榜单页。
- [ ] 增加新手引导和常见错误排查。
- [ ] 增加 PWA 添加到主屏幕能力。

#### P2：形成差异化

- [ ] 浏览器端本地制作种子。
- [ ] 一键调用本机 BT 客户端开始做种。
- [ ] 安卓端上传助手或 PWA 分享入口。
- [ ] 校园公开资料 seedbox。
- [ ] 上传者等级、徽章和贡献页。
- [ ] 面向社团/课程的官方资源发布入口。

### 六、建议的第一个版本范围

如果时间有限，第一个版本只做这些：

- 移动端可用的首页、列表页、详情页。
- 拖拽上传 `.torrent`。
- 上传完成后的做种向导。
- 新版 30 秒上传教程。
- 每周专题合集。
- 简单统计：上传数、下载数、有效做种数、移动端访问占比。

这套组合能同时解决两个关键问题：新人更敢用，上传者更不容易卡死。

### 七、衡量是否复兴成功

- 上传转化率：进入上传页的人，有多少最终成功发布并开始做种。
- 有效上传数：每周新增且至少保持 24 小时可下载的资源数量。
- 移动端占比：手机/平板访问量是否明显上升。
- 死种率：列表页中无做种资源比例是否下降。
- 回访率：用户是否在一周内再次访问。
- 社区供给：是否有社团、课程、个人创作者持续发布内容。

### 八、最大风险

- **合规风险**：必须明确禁止违法违规和未授权传播内容，保留举报与处理机制。
- **维护风险**：毕业后无人维护会再次衰落，需要尽早找接班人。
- **存储/带宽风险**：如果做服务器代存文件，成本和审核压力会激增。
- **体验风险**：如果只做 App 不改上传流程，核心问题不会解决。

### 九、推荐执行顺序

1. 找管理员确认权限和边界。
2. 用 5-10 个同学测试旧流程，拿到真实失败点。
3. 先改上传页和教程。
4. 同步做移动端/PWA 首页和详情页。
5. 用专题和社团合作拉第一批内容。
6. 数据稳定后，再考虑上传助手、安卓端和 seedbox。
